## Audit-spec corrections (the "try to break it" pass earned its keep)

Of the six pre-Stage-C audit fix-specs, **two were materially wrong** and were
corrected before shipping; a follow-up agent fix for FX-072 was also caught wrong:

- **FX-071 — audit metric was backwards.** It compared our SELL *limit* to the
  best bid and skipped when the limit was far below the bid — but a SELL limit
  below the bid fills at the *better* bid price (a good dump). The correct guard
  compares the executable price to the **cost basis** and needs no extra
  `get_merged_book` RPC. Shipped the corrected version.
- **FX-072 — audit's snapshot fix was ineffective + RPC-unacceptable.** It
  snapshotted on-chain balance at each cycle top and *overwrote it every cycle*,
  so by the cycle that notices the drain the snapshot already held the
  *post*-drain value (`max(current, snapshot)` recovers nothing). It also added
  ~250 serialized `get_balance_allowance` calls/cycle ≈ 37.5 s through the 0.15 s
  rate limiter — blowing the <30 s Stage-B cycle budget on the snapshot pass
  alone.
- **FX-072 — the follow-up "invariant add-back" fix was also wrong.** It used the
  *post*-drain probe value and added `dumped` to tracked, which over-counts and
  **misses** a partial concurrent fill (counter-example: recorded 50 + missed 20,
  dump 50 → it computes drift −30 and loses the 20, where the raw sweep already
  catches it).

The shipped FX-072 fix (below) is the third, correct design.

## FX-072 mechanism + the shipped fix (technical record)

**The leak.** A dump SELL fills **on-chain** (drains D shares) but its unwind is
**not recorded the same cycle** — either `check_dump_fills` defers it, or its
phantom check (`dump_manager.py:70`, `on_exchange >= tracked-0.5`) mis-fires
because a *concurrent BUY* replenished the balance, so it skips `record_unwind`
and clears `dump_state`. In that window `tracked` overstates holdings by D. So in
`detect_fills`, `_check_buy_phantom_fill` computes `on_chain - tracked < matched`
and **zeroes the real concurrent BUY**, and the drift sweep's raw
`on_chain - tracked` also can't see it. The fill is lost. This is a multi-way
race that fools *both* the FX-037 BUY-side and FX-007 SELL-side phantom defenses
at once — which is why surgical fixes kept failing.

**The fix — a 3-gate add-back** in `_reconcile_balance_drift`. Add the drained
dump shares back to `on_chain` only when ALL three hold (else behave exactly as
before):

1. **CAPTURED** — an outstanding dump was snapshotted at cycle top
   (`capture_pre_cycle_dumps`, free/in-memory, no RPC, before `check_dump_fills`).
2. **DRAINED** — that dump order is absent from the **cycle-top `open_ids`
   snapshot** (it left the book = filled; a reprice-cancel is still in the
   snapshot, because `open_ids` is computed *before* `reprice_active_dumps`).
3. **NOT-UNWOUND** — `check_dump_fills` did **not** record the unwind this cycle
   (else `tracked` was correctly reduced; flag set on `record_unwind`).

The recovered quantity is exactly the missed fill. The catch-up still routes
through `handle_fill` with the bucketed `drift:{cid}:{side}:{bucket}` event_id, so
**FX-065** prevents double-counting and **FX-067** unwind idempotency is
untouched. **Zero added RPC.**

Verified: reproduction test fails pre-fix (`_count_fills==0`) and passes post-fix
(`==1`); false-positive guards for clean-dump (gate 3) and resting-dump (gate 2,
FX-037 preserved) hold; mutation-checked as genuine oracles.
