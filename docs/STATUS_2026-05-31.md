## What shipped

### FX-078 — oversight owns `portfolio_snapshots` + writes `total_value` (`a264d72`)
- `simple_oversight.write_portfolio_snapshot` (a) never created the table — only the
  legacy `oversight/safety_controller` did, so a fresh deploy under `simple_oversight`
  alone would be table-less — and (b) the INSERT omitted the NOT-NULL `total_value`
  column → `IntegrityError` on **every** call, swallowed by `except: log.debug`.
- **Effect:** `portfolio_snapshots` froze at **2026-05-25 08:20 UTC** (the swap from
  `oversight_agent` to `simple_oversight`). `get_wallet_peak_usd` reads
  `MAX(exchange_balance)`; an empty table → falls back to current wallet →
  `peak == wallet` → `check_kill_switch` drawdown `= 1 − w/peak = 0` → the **15 %
  drawdown kill limb never trips**. Confirmed live: `[SIMPLE_ALLOC] peak==wallet==1201.76`.
  The independent 24h-realized-loss limb (reads `unwinds`) was unaffected.
- **Fix:** `CREATE TABLE IF NOT EXISTS` (canonical 6-col schema) + INSERT `total_value`
  (= `exchange_balance`) + `try/finally` close (fixes a per-failure connection leak) +
  swallowed log `debug → warning`.
- **Test (P4 lesson):** the `test_simple_oversight` fixture had a 2-column
  `portfolio_snapshots` — diverged from the 6-column production schema — which is why
  59 green tests hid this for 6 days. Fixture realigned to the prod schema; O8
  strengthened; new O8b for the fresh-DB create path. Both fail pre-fix, pass post-fix.

### FX-080 — roll back failed DB writes so they don't wedge the shared connection (`8aa10f5`, closes FX-075)
- `BotDatabase` uses one thread-local connection per process. ~30 write methods did
  `execute → commit` inside `try / except: log.debug` with **no rollback**, so a single
  failed write (e.g. a commit that lost the WAL write lock) left an **open transaction**
  on the shared connection.
- **Effect (oversight process):** the open transaction held the WAL write lock →
  `wallet_reconcile_history` froze at the **2026-05-29 04:22** restart+1 cycle (logs
  `[WALLET_RECONCILE] ok` every cycle but **0 rows persisted**), the ROI cache upsert
  failed `database is locked` **every cycle**, and the WAL grew to **95 MB** with the
  main DB file frozen ~2 days. The farmer's own connection stayed healthy
  (`capital_committed_snapshots` current), so the wedge was oversight-side. Desync
  **detection + Discord alert** were unaffected (log branch, not the row write).
- **Fix:** new `_rollback_quiet()` (rolls back the thread-local connection's pending
  transaction; no-op when clean, never raises) called in the `except` of **all 30 write
  methods**; swallowed logs `debug → warning` (truthful, FX-054 class). Reads
  (`get_*`/`load_*`/`sum_*`) untouched — they open no write transaction. **Hardens the
  live farmer too** (its writers had the same latent wedge), which matters at cutover.
- **Test:** `TestRollbackQuietFX080` — wedge (open transaction) → `_rollback_quiet`
  clears it → connection recovers and a normal write commits; + clean no-op case.

### FX-079 — dissolved (not a bug)
Empirically, bare `sqlite3.connect()` already sets `busy_timeout = 5000 ms` (Python's
`timeout` default is 5.0 s), so the proposed "add busy_timeout to the ROI tracker's raw
connects" was a no-op. The every-cycle `[ROI_TRACKER] … database is locked` was a
**symptom of the FX-080 wedge** (a held WAL write lock), not a missing timeout. Expect
it to clear once FX-080 lands; the now-truthful logging will surface any residual.
